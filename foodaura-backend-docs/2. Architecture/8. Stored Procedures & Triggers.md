# Stored Procedures & Triggers

All domain logic lives in MariaDB stored procedures. Both the Go backend and the AI service call these directly. Neither re-implements logic in application code.

Naming convention: `sp_<feature>_<action>` for procedures, `trg_<table>_<timing>_<event>` for triggers.

---

## Input Validation

Procedures validate inputs early and raise `SIGNAL SQLSTATE '45000'` with a descriptive message on failure. The backend maps this to a 400 Bad Request response. Rules per procedure:

| Procedure | Validation rules |
|---|---|
| `sp_auth_create_user` | `email` not null/empty, must match `%@%.%`; `password_hash` not null/empty |
| `sp_household_create` | `name` not null, `TRIM(name)` not empty |
| `sp_household_invite` | `email` must match `%@%.%`; `expires_at` must be in the future |
| `sp_profile_upsert` | `age` 1–120; `weight_kg` 10–500; `height_cm` 50–300 |
| `sp_mealplan_notify_ready` | `meal_plan_id` must reference an existing meal plan |

Constraints enforced by the DB schema (NOT NULL columns, UNIQUE keys, ENUM column types, foreign keys) are not duplicated in procedure validation — the schema is the last line of defence, procedures give early, readable errors.

---

## Notification Lifecycle

Notifications are **never deleted**. Once created they persist and are only ever transitioned to `is_read = TRUE` via `sp_notification_mark_read` or `sp_notification_mark_all_read`. This ensures a complete audit trail of household events for each user.

## Triggers

### `trg_nutritional_profiles_after_upsert`
**Event:** `AFTER INSERT OR UPDATE ON nutritional_profiles`
**Action:** Calls `sp_profile_calculate_targets(user_id)` to recalculate and store the user's nutritional targets whenever their profile is created or changed.
**Why a trigger:** Ensures targets are always in sync with the profile without requiring the caller to remember to recalculate.

## Stored Procedures

### Reference Data

#### `sp_goals_list`
**Purpose:** Return all available goal options ordered by `sort_order`. Called by `GET /onboarding` and `GET /profile/edit` to populate pickers. The frontend never hardcodes goal values or labels — it uses this result.
**Parameters:** —
**Returns:** `value`, `label`, `icon` for each row in the `goals` table.

#### `sp_diet_types_list`
**Purpose:** Return all available diet type options ordered by `sort_order`. Same usage pattern as `sp_goals_list`.
**Parameters:** —
**Returns:** `value`, `label` for each row in the `diet_types` table.

### AI

#### `sp_ai_get_household_profiles`
**Purpose:** Return the nutritional profiles and targets for all members of a household. The only procedure the AI service uses to access user data — email and all other personally identifying information are excluded. Called by the AI service when generating a meal plan.
**Parameters:** `household_id UUID`
**Returns:** For each household member: `user_id`, `age`, `weight_kg`, `height_cm`, `activity_level`, `goal`, `calories`, `protein_g`, `carbs_g`, `fat_g`

### Auth

#### `sp_auth_create_user`
**Purpose:** Create a new user account.
**Parameters:** `email VARCHAR`, `display_name VARCHAR`, `password_hash VARCHAR`
**Returns:** `user_id UUID`

#### `sp_auth_get_user_by_email`
**Purpose:** Fetch a user record by email for login verification.
**Parameters:** `email VARCHAR`
**Returns:** `id`, `email`, `display_name`, `password_hash`

#### `sp_auth_create_session`
**Purpose:** Persist a new session after successful login.
**Parameters:** `user_id UUID`, `token_hash VARCHAR`, `expires_at DATETIME`
**Returns:** `session_id UUID`

#### `sp_auth_get_session`
**Purpose:** Validate a session token, extend its expiry by 30 days (rolling), and return the associated user. Used on every authenticated request.
**Parameters:** `token_hash VARCHAR`
**Returns:** `user_id`, `new_expires_at` — empty result if not found or expired.

#### `sp_auth_delete_session`
**Purpose:** Delete a session on logout.
**Parameters:** `token_hash VARCHAR`
**Returns:** —

### Household

#### `sp_household_create`
**Purpose:** Create a new household and add the creator as both owner and first member.
**Parameters:** `name VARCHAR`, `owner_user_id UUID`
**Returns:** `household_id UUID`

#### `sp_household_get`
**Purpose:** Fetch household details. Scoped — only succeeds if the requesting user is a member.
**Parameters:** `household_id UUID`, `requesting_user_id UUID`
**Returns:** Household row + member list. Each member row includes `display_name`, `initials` (first letter of each word in display_name, up to 2 chars), and `avatar_tint` (derived from join order: owner → `'p'`; alternates `'s'`/`'p'` for subsequent members).

#### `sp_household_invite`
**Purpose:** Create a household invitation for the given email address. If the invitee already has a Foodaura account, creates a `household_invitation_received` notification for them immediately.
**Parameters:** `household_id UUID`, `invited_by_user_id UUID`, `email VARCHAR`, `token_hash VARCHAR`, `expires_at DATETIME`
**Returns:** `invitation_id UUID`

#### `sp_household_accept_invitation`
**Purpose:** Mark an invitation as accepted, add the user to the household, and create a `household_invitation_accepted` notification for the inviter. The invitee was already notified when `sp_household_invite` was called.
**Parameters:** `token_hash VARCHAR`, `accepting_user_id UUID`
**Returns:** `household_id UUID`

#### `sp_household_remove_member`
**Purpose:** Remove a member from a household. Only callable by the household owner. Creates a `household_member_removed` notification for the removed user.
**Parameters:** `household_id UUID`, `target_user_id UUID`, `requesting_user_id UUID`
**Returns:** —

#### `sp_household_leave`
**Purpose:** Allow a non-owner member to leave a household. Creates a `household_member_left` notification for the owner.
**Parameters:** `household_id UUID`, `user_id UUID`
**Returns:** —

### Profile & Targets

#### `sp_profile_upsert`
**Purpose:** Create or update a user's nutritional profile. The `trg_nutritional_profiles_after_upsert` trigger fires automatically after this to recalculate targets.
**Parameters:** `user_id UUID`, `biological_sex ENUM`, `age TINYINT`, `weight_kg DECIMAL`, `height_cm DECIMAL`, `activity_level ENUM`, `goal VARCHAR(50)`, `diet_type VARCHAR(50)`
**Returns:** —
**Notes:** `goal` and `diet_type` are validated by the FK constraints on `nutritional_profiles`, not by the procedure itself.

#### `sp_profile_get`
**Purpose:** Fetch a user's nutritional profile.
**Parameters:** `user_id UUID`
**Returns:** Full `nutritional_profiles` row.

#### `sp_profile_calculate_targets`
**Purpose:** Calculate nutritional targets from the user's profile and write them to `nutritional_targets`. Called by the trigger — not called directly by the backend.
**Parameters:** `user_id UUID`
**Returns:** —
**Notes:** Uses Mifflin-St Jeor formula. Full calculation defined in [Nutritional Calculation](11.%20Nutritional%20Calculation.md). Inputs: biological_sex, age, weight_kg, height_cm, activity_level, goal.

#### `sp_targets_get`
**Purpose:** Fetch a user's stored nutritional targets.
**Parameters:** `user_id UUID`
**Returns:** Full `nutritional_targets` row.

### Recipes

#### `sp_recipe_find`
**Purpose:** Return a filtered list of recipes. Default filter excludes recipes containing the user's disliked ingredients, recipes the user has marked as disliked, and recipes structurally incompatible with the user's `diet_type` (e.g. meat-containing recipes are excluded for `vegan`/`vegetarian`). Additional filters: tag, cuisine, cook time, free-text search.
**Parameters:** `user_id UUID`, `tag_id UUID (nullable)`, `cuisine VARCHAR (nullable)`, `max_cook_time_minutes SMALLINT (nullable)`, `search_term VARCHAR (nullable)`, `limit INT`, `offset INT`
**Returns:** List of recipe summary rows: `id`, `title`, `cuisine`, `cook_time_minutes`, `servings_base`, `calories`, `protein_g`, `carbs_g`, `fat_g` + associated tags.

#### `sp_recipe_get`
**Purpose:** Fetch full recipe detail including ingredients, steps, and tags.
**Parameters:** `recipe_id UUID`
**Returns:** Recipe row + `recipe_ingredients` rows + `recipe_steps` rows (ordered by `step_number`) + associated tags.

#### `sp_recipe_scale`
**Purpose:** Return a recipe's ingredient quantities scaled to the participants of a given meal slot. Scaling respects each participant's nutritional targets rather than multiplying servings evenly.
**Parameters:** `recipe_id UUID`, `meal_slot_id UUID`
**Returns:** List of `(ingredient_name, scaled_quantity, unit, category)`.

### Meal Plans

#### `sp_mealplan_get_or_create`
**Purpose:** Return the meal plan for the given household and week, creating it (with empty slots) if it does not exist. Slots are created for all combinations of day (Mon–Sun) and meal type (breakfast, lunch, dinner). All household members are added as default participants on every slot.
**Parameters:** `household_id UUID`, `week_start_date DATE`, `requesting_user_id UUID`
**Returns:** `meal_plan_id UUID`

#### `sp_mealplan_get`
**Purpose:** Fetch a full meal plan including all slots, assigned recipes, and participants per slot. Scoped to the requesting user's household.
**Parameters:** `meal_plan_id UUID`, `requesting_user_id UUID`
**Returns:** Meal plan row + slot rows + participant rows per slot.

#### `sp_mealplan_assign_recipe`
**Purpose:** Assign a recipe to a meal slot and set the participant list. Replaces any existing assignment on the slot. Triggers shopping list regeneration for the parent meal plan.
**Parameters:** `meal_slot_id UUID`, `recipe_id UUID`, `participant_user_ids ARRAY/JSON`
**Returns:** —

#### `sp_mealplan_clear_slot`
**Purpose:** Remove the recipe and participants from a meal slot, leaving it empty.
**Parameters:** `meal_slot_id UUID`
**Returns:** —

#### `sp_mealplan_get_history`
**Purpose:** Return a list of past meal plans for a household, ordered by week descending. Excludes the current week.
**Parameters:** `household_id UUID`, `requesting_user_id UUID`
**Returns:** List of `(meal_plan_id, week_start_date)`.

#### `sp_mealplan_copy`
**Purpose:** Copy all slots and participants from a past meal plan into a new meal plan for a given week. Used for reusing a previous week.
**Parameters:** `source_meal_plan_id UUID`, `target_week_start_date DATE`, `requesting_user_id UUID`
**Returns:** `new_meal_plan_id UUID`

#### `sp_mealplan_notify_ready`
**Purpose:** Create a `meal_plan_ready` notification for the user who requested AI generation. Called by the backend after the AI service returns and the plan has been saved to the database.
**Parameters:** `meal_plan_id UUID`, `user_id UUID`
**Returns:** —

### Shopping List

#### `sp_shoppinglist_derive`
**Purpose:** Generate or regenerate the shopping list for a meal plan. For each assigned meal slot, calls `sp_recipe_scale` to obtain participant-scaled quantities, then groups by `(ingredient_name, unit)` and sums. Preserves `is_checked` state for items that still appear after regeneration. Deletes items that no longer appear.
**Parameters:** `meal_plan_id UUID`
**Returns:** —
**Note:** Quantities in `shopping_list_items` are always the scaled total — not the base recipe quantities.

#### `sp_shoppinglist_get`
**Purpose:** Fetch the shopping list for a meal plan, ordered by category then ingredient name. Scoped to the requesting user's household.
**Parameters:** `meal_plan_id UUID`, `requesting_user_id UUID`
**Returns:** List of `shopping_list_items` rows.

#### `sp_shoppinglist_toggle_item`
**Purpose:** Set the `is_checked` state of a shopping list item.
**Parameters:** `item_id UUID`, `is_checked BOOLEAN`
**Returns:** —

### Notifications

#### `sp_notification_create`
**Purpose:** Insert a notification row for a user. Called internally by other stored procedures when a notifiable event occurs (e.g. `sp_household_accept_invitation`, `sp_household_leave`).
**Parameters:** `user_id UUID`, `type ENUM`, `title VARCHAR`, `body TEXT`, `reference_type VARCHAR (nullable)`, `reference_id UUID (nullable)`
**Returns:** `notification_id UUID`

#### `sp_notification_get_all`
**Purpose:** Fetch all notifications for a user, ordered by `is_read ASC, created_at DESC` (unread first).
**Parameters:** `user_id UUID`
**Returns:** List of `notifications` rows.
**Bucketing (Go handler):** The controller splits the flat result into `Today` (created_at >= today's midnight) and `Week` (created_at >= this week's Monday, excluding today). Notifications older than the current week are shown in `Week` until a separate history view is built.

#### `sp_notification_mark_read`
**Purpose:** Mark a single notification as read. Only succeeds if the notification belongs to the requesting user.
**Parameters:** `notification_id UUID`, `user_id UUID`
**Returns:** —

#### `sp_notification_mark_all_read`
**Purpose:** Mark all unread notifications for a user as read.
**Parameters:** `user_id UUID`
**Returns:** —
