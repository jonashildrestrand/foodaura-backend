# Module Boundaries

## Backend (Go) — MVC

```
internal/
  controller/     ← HTTP handlers; parse input, call model, render template
  model/          ← one file per feature; calls stored procedures; returns data structs
  view/           ← html/template files; no logic
```

### Controllers (`internal/controller/`)

One file per feature. Responsibilities: parse and validate HTTP input, call the relevant model function, select a template, and render the response. No business logic.

`internal/controller/helpers.go` provides shared utilities: `serverError(w, r, msg, err)` logs the error with `slog` and renders the 500 error page; `NotFoundHandler()` returns an `http.HandlerFunc` for unmatched routes; `InitErrorRenderer(r)` wires in the `view.Renderer` at startup. `buildBaseVM` fetches sidebar data (current user, unread count) for authenticated pages.

| File | Feature |
|---|---|
| `auth.go` | Login, logout, session management |
| `user.go` | `POST /users` — single-transaction user registration (creates user, household, profile, session) |
| `onboarding.go` | `GET /onboarding` — fetches goals and diet types from DB, passes pre-serialized JSON to the template; Alpine.js controls all step visibility client-side |
| `household.go` | Household management, member management, invitation handling |
| `profile.go` | Profile setup and editing |
| `recipe.go` | Recipe discovery and detail |
| `mealplan.go` | Meal plan display, slot assignment, slot clearing |
| `shoppinglist.go` | Shopping list display and item toggling |
| `notification.go` | Notification list, mark-read |

### Models (`internal/model/`)

One file per feature. Responsibilities: call the appropriate stored procedure, scan results into Go structs, return them to the controller. No HTTP awareness.

| File | Stored procedures called |
|---|---|
| `auth.go` | `sp_auth_get_user`, `sp_auth_get_user_by_email`, `sp_auth_create_user`, `sp_auth_create_session`, `sp_auth_get_session`, `sp_auth_delete_session` |
| `household.go` | `sp_household_find_by_user`, `sp_household_create`, `sp_household_get`, `sp_household_invite`, `sp_household_accept_invitation`, `sp_household_remove_member`, `sp_household_leave` |
| `profile.go` | `sp_profile_upsert`, `sp_profile_get`, `sp_targets_get` |
| `recipe.go` | `sp_recipe_find`, `sp_recipe_get`, `sp_recipe_scale`, `sp_tag_list`, `sp_preference_set_recipe` |
| `mealplan.go` | `sp_mealplan_get_or_create`, `sp_mealplan_get`, `sp_mealplan_assign_recipe`, `sp_mealplan_clear_slot`, `sp_mealplan_get_history` |
| `shoppinglist.go` | `sp_shoppinglist_derive`, `sp_shoppinglist_get`, `sp_shoppinglist_toggle_item` |
| `notification.go` | `sp_notification_get_all`, `sp_notification_unread_count`, `sp_notification_mark_read`, `sp_notification_mark_all_read` |
| `reference.go` | `sp_goals_list`, `sp_diet_types_list` |

**Error pages** — `templates/pages/error.gohtml` is a no-sidebar, centered page (same layout pattern as login). It displays the numeric status code, a human title, and a short message using the app's design tokens. The 404 handler is registered via `r.NotFound` in `main.go`; 500s are triggered from `serverError` in `internal/controller/helpers.go`. Both go through `Renderer.RenderError`.

### Middleware (`internal/middleware/`)

| File | Responsibility |
|---|---|
| `auth.go` | Validates the `session` cookie on every protected route; injects `userID` into the request context |
| `logging.go` | Logs every request with method, path, status code, and latency using `log/slog`; 5xx → Error, 4xx → Warn, 2xx/3xx → Info |

### Views (`internal/view/` and `templates/`)

Go `html/template` files. Receive plain structs from the controller (`internal/vm/`). Template execution is buffered — rendered to a `bytes.Buffer` before writing to the response, so a mid-template error produces a proper 500 rather than a partial blank page. No business logic in templates.

`Renderer` exposes two public methods:
- `Render(w, pageFile, data)` — renders any page with HTTP 200.
- `RenderError(w, status)` — builds an `ErrorVM` from the status code and renders `pages/error.gohtml` with the correct status. Falls back to `http.Error` plain text if the template itself fails to render.

Both share an internal `renderBuffer` method that parses and executes the template into a `bytes.Buffer` before touching the `ResponseWriter`, keeping the header/status write after the point where failure is possible.

## Database — Stored Procedures

Domain logic lives in MariaDB stored procedures, grouped by feature. Both the backend and the AI service call these procedures directly, ensuring the same rules are applied regardless of which consumer initiates the operation.

| Feature | Procedures | Domain logic enforced |
|---|---|---|
| **Reference** | `sp_goals_list`, `sp_diet_types_list` | Serves the DB-authoritative goal and diet type options to the frontend |
| **Auth** | `sp_auth_*` | User creation, credential lookup, session lifecycle |
| **Household** | `sp_household_*` | Membership rules, invitation lifecycle, household scoping |
| **Profile** | `sp_profile_upsert`, `sp_profile_get`, `sp_targets_get` | Nutritional target calculation (Mifflin-St Jeor); goal and diet validation via FK |
| **Recipe** | `sp_recipe_find`, `sp_recipe_get`, `sp_recipe_scale` | Filtering by diet type and dislikes; ingredient scaling to participant targets |
| **Meal Plan** | `sp_mealplan_*` | Slot structure, participant defaults, shopping list regeneration trigger |
| **Shopping List** | `sp_shoppinglist_derive`, `sp_shoppinglist_get`, `sp_shoppinglist_toggle_item` | Ingredient consolidation and quantity summing across all assigned slots |
| **Notifications** | `sp_notification_*` | Event creation, read state, unread count |
