# Database Migrations

Migrations are managed with **Flyway**, running as a one-shot Docker container. Flyway applies all pending migrations against MariaDB on startup and exits. The backend container only starts after Flyway completes successfully. The Go backend has no involvement in database provisioning.

---

## File Structure

```
db/
  migrations/
    V001__create_tables.sql             ← versioned: table DDL
    V002__grants.sql                    ← versioned: DB user grants
    R__01_procedures_ai.sql             ← repeatable: AI stored procedures
    R__02_procedures_auth.sql           ← repeatable: auth stored procedures
    R__03_procedures_household.sql      ← repeatable: household stored procedures
    R__04_procedures_profile.sql        ← repeatable: profile stored procedures
    R__05_procedures_recipe.sql         ← repeatable: recipe stored procedures
    R__06_procedures_mealplan.sql       ← repeatable: meal plan stored procedures
    R__07_procedures_shoppinglist.sql   ← repeatable: shopping list stored procedures
    R__08_procedures_notification.sql   ← repeatable: notification stored procedures

  seed/

  test/
    helpers.sql
    run_tests.sh
    ...
```

## Versioned vs Repeatable Migrations

Flyway supports two migration types, and this project uses both deliberately.

**Versioned (`V__`)** — run once, never again. Used for table DDL and grants. Once `V001__create_tables.sql` is applied, it is never modified — schema changes (new columns, new tables) go in a new versioned file (e.g. `V003__add_column_to_recipes.sql`).

**Repeatable (`R__`)** — run whenever their content changes. Used for stored procedures and triggers. When a stored procedure is modified, update the relevant `R__` file and Flyway will re-apply it on next startup. No new versioned file needed. Repeatable migrations always run after all versioned migrations, in alphabetical order — the numeric prefix ensures a consistent order.

This split matches the nature of the changes:
- Table structure changes need an audit trail — versioned
- Stored procedures are effectively `CREATE OR REPLACE` operations — repeatable

## Stored Procedure Syntax

Flyway uses `;` as its default statement separator, which conflicts with the `;` used inside `BEGIN...END` blocks. Use `DELIMITER $$` to work around this:

```sql
-- R__04_procedures_profile.sql

DELIMITER $$

DROP PROCEDURE IF EXISTS sp_profile_upsert $$
CREATE PROCEDURE sp_profile_upsert(
  IN p_user_id    CHAR(36),
  IN p_bio_sex    ENUM('male', 'female'),
  IN p_age        TINYINT UNSIGNED,
  IN p_weight_kg  DECIMAL(5,2),
  IN p_height_cm  DECIMAL(5,2),
  IN p_activity   ENUM('sedentary','light','moderate','active','very_active'),
  IN p_goal       ENUM('lose_weight','maintain','build_muscle','eat_better')
)
BEGIN
  INSERT INTO nutritional_profiles (user_id, biological_sex, age, weight_kg, height_cm, activity_level, goal)
  VALUES (p_user_id, p_bio_sex, p_age, p_weight_kg, p_height_cm, p_activity, p_goal)
  ON DUPLICATE KEY UPDATE
    biological_sex = p_bio_sex,
    age            = p_age,
    weight_kg      = p_weight_kg,
    height_cm      = p_height_cm,
    activity_level = p_activity,
    goal           = p_goal;
END $$

DELIMITER ;
```

## Docker Compose Integration

```yaml
services:
  mariadb:
    image: mariadb:11
    healthcheck:
      test: ["CMD", "healthcheck.sh", "--connect", "--innodb_initialized"]
      interval: 5s
      retries: 10

  flyway:
    image: flyway/flyway:11-alpine
    command: migrate
    volumes:
      - ./db/migrations:/flyway/sql
    depends_on:
      mariadb:
        condition: service_healthy
    environment:
      FLYWAY_URL: jdbc:mariadb://mariadb:3306/foodaura
      FLYWAY_USER: root
      FLYWAY_PASSWORD: ${MARIADB_ROOT_PASSWORD}
      FLYWAY_LOCATIONS: filesystem:/flyway/sql

  backend:
    depends_on:
      flyway:
        condition: service_completed_successfully
```

Flyway exits with code `0` on success. Docker Compose treats `service_completed_successfully` as a gate — the backend will not start if Flyway fails.

## Test Database

The test suite runs against a separate `foodaura_test` database. `run_tests.sh` provisions it by running the same migration files directly via the `mysql` client — no Flyway involvement for tests:

```bash
#!/bin/bash
set -e
DB="foodaura_test"
MYSQL="mysql -u root -p${MARIADB_ROOT_PASSWORD}"

$MYSQL -e "DROP DATABASE IF EXISTS $DB; CREATE DATABASE $DB;"

# Apply schema in the same order Flyway would
for f in db/migrations/V*.sql; do $MYSQL $DB < "$f"; done
for f in db/migrations/R__*.sql; do $MYSQL $DB < "$f"; done

# Load MyTAP and fixtures
$MYSQL $DB < db/test/mytap.sql
$MYSQL $DB < db/test/helpers.sql

# Run tests
for f in db/test/*_test.sql; do
  echo "--- $f"
  $MYSQL $DB < "$f"
done
```

## Adding a Migration

| Change type | Action |
|---|---|
| New table or column | Add a new `V00N__description.sql` file |
| Modified stored procedure | Update the relevant `R__0N_procedures_*.sql` file |
| Modified trigger | Update `R__09_triggers.sql` |
| New stored procedure | Add it to the relevant `R__` file |
| New DB user grant | Add a new `V00N__grants.sql` file |
