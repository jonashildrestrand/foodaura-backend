# Module Boundaries

## Backend (Go) — MVC

```
internal/
  controller/     ← HTTP handlers; parse input, call model, render template
  model/          ← one file per feature; calls stored procedures; returns data structs
  view/           ← html/template files; no logic
```

### Controllers (`internal/controller/`)

One file per feature. Responsibilities: parse and validate HTTP input, call the relevant model function, select a template, and render the response. No business logic.

| File | Feature |
|---|---|
| `auth.go` | Login, logout, session management |
| `user.go` | `POST /users` — single-transaction user registration (creates user, household, profile, session) |
| `onboarding.go` | Single `GET /onboarding` handler — renders all steps in one page; Alpine.js controls which step is visible; no DB writes |
| `household.go` | Household management, member management |
| `profile.go` | Profile updates |
| `recipe.go` | Recipe discovery and detail |
| `mealplan.go` | Manual and AI-assisted meal plan management |
| `shoppinglist.go` | Shopping list generation and display |

### Models (`internal/model/`)

One file per feature. Responsibilities: open a database connection, call the appropriate stored procedure, scan results into Go structs, return them to the controller. No HTTP awareness.

| File | Stored procedures called |
|---|---|
| `household.go` | `sp_create_household`, `sp_invite_member`, `sp_get_household` |
| `profile.go` | `sp_upsert_profile`, `sp_get_profile` |
| `recipe.go` | `sp_find_recipes`, `sp_get_recipe`, `sp_scale_recipe` |
| `mealplan.go` | `sp_create_meal_plan`, `sp_update_meal_plan`, `sp_get_meal_plan` |
| `shoppinglist.go` | `sp_derive_shopping_list`, `sp_get_shopping_list` |

### Views (`internal/view/` or `templates/`)

Go `html/template` files. Receive plain structs from the controller. No logic beyond template conditionals and iteration.

---

## Database — Stored Procedures

Domain logic lives in MariaDB stored procedures, grouped by feature. Both the backend and the AI service call these procedures directly, ensuring the same rules are applied regardless of which consumer initiates the operation.

| Feature | Procedures | Domain logic enforced |
|---|---|---|
| **Household** | `sp_create_household`, `sp_invite_member`, `sp_get_household` | Household scoping; membership rules |
| **Profile** | `sp_upsert_profile`, `sp_get_profile` | Goal and constraint validation |
| **Recipe** | `sp_find_recipes`, `sp_get_recipe`, `sp_scale_recipe` | Filtering by profile and goals; ingredient scaling to serving count |
| **Meal Plan** | `sp_create_meal_plan`, `sp_update_meal_plan`, `sp_get_meal_plan` | Per-member and household-wide scheduling; AI-generated plan storage |
| **Shopping List** | `sp_derive_shopping_list`, `sp_get_shopping_list` | Ingredient consolidation and unit normalisation across meal plan entries |
