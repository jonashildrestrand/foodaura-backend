# Database Testing

Stored procedures are tested with **MyTAP** — a unit testing framework for MariaDB that produces TAP (Test Anything Protocol) output. Tests are plain SQL files and run against a dedicated `foodaura_test` database that is provisioned using the same migration files as production.

---

## File Structure

```
db/
  test/
    mytap.sql              ← MyTAP installation (downloaded once, committed to repo)
    helpers.sql            ← standing fixtures shared across all test files
    01_auth_test.sql
    02_household_test.sql
    03_profile_test.sql
    04_recipe_test.sql
    05_mealplan_test.sql
    06_shoppinglist_test.sql
    07_notification_test.sql
    08_validation_test.sql ← input validation and rejection tests (all procedures)
    run_tests.sh           ← provisions test DB and runs all test files
```

---

## Running Tests

```bash
./db/test/run_tests.sh
```

`run_tests.sh` provisions a fresh `foodaura_test` database, applies all migrations in the same order Flyway would, installs MyTAP, loads fixtures, and runs each test file:

```bash
#!/bin/bash
set -e

DB="foodaura_test"
MYSQL="mysql -u root -p${MARIADB_ROOT_PASSWORD}"

# Fresh database every run
$MYSQL -e "DROP DATABASE IF EXISTS $DB; CREATE DATABASE $DB;"

# Apply schema — same order as Flyway
for f in db/migrations/V*.sql; do $MYSQL $DB < "$f"; done
for f in db/migrations/R__*.sql; do $MYSQL $DB < "$f"; done

# Install MyTAP and load fixtures
$MYSQL $DB < db/test/mytap.sql
$MYSQL $DB < db/test/helpers.sql

# Run test files
for f in db/test/*_test.sql; do
  echo "--- $f"
  $MYSQL $DB < "$f"
done
```

## Test Isolation

Each test section uses **explicit DELETE / mark-as-read cleanup** rather than `ROLLBACK`. `ROLLBACK` cannot be used because MyTAP writes assertion results to its own table (`tap.__tresults__`) inside the same connection — a rollback would undo those rows and break the TAP output.

The cleanup pattern:
1. **Data rows**: `DELETE FROM <table> WHERE <test-specific condition>` after each test section.
2. **Notifications**: `CALL sp_notification_mark_all_read(@user_id)` — notifications are never hard-deleted in the application; tests follow the same rule.

The standing fixtures in `helpers.sql` are created once before the test files run and persist for the entire test session — they are the baseline state every test starts from.

## Fixtures (`helpers.sql`)

Creates the minimum standing data all test files can reference without repeating setup:

```sql
-- One user per common test scenario
CALL sp_auth_create_user('owner@test.com', '$2b$12$testhash');
CALL sp_auth_create_user('member@test.com', '$2b$12$testhash');
CALL sp_auth_create_user('outsider@test.com', '$2b$12$testhash');

-- One household owned by owner@test.com with member@test.com as a member
SET @owner_id  = (SELECT id FROM users WHERE email = 'owner@test.com');
SET @member_id = (SELECT id FROM users WHERE email = 'member@test.com');
CALL sp_household_create('Test Household', @owner_id);
SET @household_id = (SELECT id FROM households WHERE owner_user_id = @owner_id);
CALL sp_household_accept_invitation(...); -- member joins
```

## Test File Structure

Each test file covers one feature group and follows the same pattern:

```sql
-- 03_profile_test.sql
SELECT tap.plan(12);

-- Test: targets calculated correctly for male, build_muscle
CALL sp_profile_upsert(@owner_id, 'male', 30, 80.0, 180.0, 'moderate', 'build_muscle');

SELECT calories  INTO @cal  FROM nutritional_targets WHERE user_id = @owner_id;
SELECT protein_g INTO @pro  FROM nutritional_targets WHERE user_id = @owner_id;
SELECT fat_g     INTO @fat  FROM nutritional_targets WHERE user_id = @owner_id;
SELECT carbs_g   INTO @carb FROM nutritional_targets WHERE user_id = @owner_id;

SELECT tap.eq(@cal,  '3059', 'build_muscle: calories correct');
SELECT tap.eq(@pro,  '160',  'build_muscle: protein is 2.0g/kg');
SELECT tap.eq(@fat,  '85',   'build_muscle: fat is 25% of calories');
SELECT tap.eq(@carb, '414',  'build_muscle: carbs fill remainder');

CALL tap.finish();
```

### Validating that procedures reject bad input

To test that a `SIGNAL SQLSTATE '45000'` is raised, create a tiny single-use helper procedure with a `CONTINUE HANDLER`, call it, then drop it. The `DELIMITER $$` block must surround the `CREATE PROCEDURE` statement.

```sql
DELIMITER $$
CREATE PROCEDURE _t() BEGIN
  DECLARE CONTINUE HANDLER FOR SQLSTATE '45000' SET @err = 1;
  SET @err = 0;
  CALL sp_that_should_fail(bad_input);
END$$
DELIMITER ;
CALL _t(); DROP PROCEDURE _t;
SELECT tap.ok(@err = 1, 'procedure: rejects bad input');
```

All validation tests live in `08_validation_test.sql`.

## What to Test per Feature

| Feature | Key scenarios to cover |
|---|---|
| **Auth** | Create user, get user by email, create/validate/delete session, expired session returns empty |
| **Household** | Create household (owner added as member), invite, accept invitation, remove member (notifies user), leave (notifies owner), outsider cannot access |
| **Profile & Targets** | Targets calculated on create, targets recalculate on update, each goal/sex combination produces correct values |
| **Recipes** | Find respects preference filters, disliked recipes excluded, ingredient dislikes excluded, scale returns correct quantities per participant |
| **Meal Plans** | Slots created for all days/meal types on plan creation, all members added as default participants, assign recipe, clear slot, copy plan preserves slots |
| **Shopping List** | Derived list consolidates same ingredient+unit across slots, checked state preserved on regeneration, cleared items removed |
| **Notifications** | Correct notification created for each event type, mark read, mark all read, outsider cannot mark another user's notification read |
| **Validation** | Each procedure with input guards: null/empty, out-of-range, and format errors all raise SQLSTATE '45000' |

## MyTAP Installation

MyTAP is not installed via a package manager — download the source and commit `mytap.sql` to the repo:

```bash
curl -L https://github.com/hepabolu/mytap/raw/main/mytap.sql -o db/test/mytap.sql
```

Commit `mytap.sql` so the test suite has no external dependencies at runtime.
