# Design Decisions

## Domain logic in stored procedures

**Decision:** Business rules (recipe scaling, shopping list derivation, meal plan generation, nutritional calculations) are implemented as MariaDB stored procedures. The Go backend and the AI service both call these procedures directly.

**Why:** Both consumers communicate with the database directly — the AI service is not routed through the backend. If domain logic lived in the Go application layer, the AI service would have no way to benefit from it and rules would need to be duplicated or skipped entirely. Stored procedures solve this: the logic is defined once, in the database, and applies to every caller regardless of which service initiates the operation. It also means a single place to test, version, and correct business rules when they change.

**Trade-off:** Business logic in SQL is harder to unit-test in isolation than Go code, and stored procedures require schema migrations to change. This is acceptable because the domain rules here — scaling, consolidation, filtering — are stable once defined and are naturally expressed as set-based operations on the data the database already owns.

---

## MVC for the Go backend

**Decision:** The backend follows a straightforward MVC structure. Controllers handle HTTP, models call stored procedures and return data structs, views are Go templates.

**Why:** With domain logic residing in stored procedures, there is no need for a dedicated application layer or repository abstractions in Go. The model layer's job is thin: call the right procedure, scan the result, and hand it to the controller. MVC maps directly onto that without introducing unnecessary indirection.

---

## AI service communicates with the database directly

**Decision:** The AI service calls MariaDB stored procedures directly rather than going through the backend's HTTP API.

**Why:** Routing AI operations through the backend HTTP API would add latency, require the AI service to authenticate as a client, and couple the AI service's capabilities to whatever the backend chooses to expose. Direct database access lets the AI service read any data it needs and invoke any stored procedure, while stored procedures still enforce domain rules. This also means the AI service is not blocked by backend API design decisions and can evolve independently.

**Trade-off:** The backend is no longer the single entry point for all writes. Domain invariants must be enforced at the stored procedure level rather than centrally in Go application code. This is intentional — see the stored procedures decision above.

## Notification table designed for future external delivery

**Decision:** Notifications are written to a `notifications` table by the backend. A separate `notification_dispatches` table tracks external delivery per channel. The backend owns neither table beyond writing the notification row — it does not send email or SMS.

**Why:** Delivery channels (email, SMS, push) are a separate concern from generating the notification event. Coupling them into the backend would require the backend to know about SMTP servers, Twilio, etc. — none of which are in scope now. By writing only to `notifications`, the backend stays simple. A future notification service can poll the table, dispatch via whatever channel it supports, and record the result in `notification_dispatches` — without any changes to the backend.

**Trade-off:** Notifications are not real-time. A polling-based service introduces delivery lag. This is acceptable for the events in scope (invitations, member changes, meal plan ready) — none are time-critical at the second level.

## Database access control through two DB users and stored procedures

**Decision:** Two MariaDB users are provisioned — `foodaura_backend` and `foodaura_ai`. Neither has direct `SELECT`, `INSERT`, `UPDATE`, or `DELETE` on any table. Both access data exclusively through stored procedures, which run with `SQL SECURITY DEFINER` and enforce row and column scoping internally.

**Why:** MariaDB has no native row-level security. Storing business logic in procedures and revoking direct table access from application users is the standard pattern for enforcing access control at the database layer. It means a bug or misconfiguration in the application code cannot result in cross-household data leakage — the DB user simply doesn't have the privilege to construct such a query. Column-level security for the AI service is handled by `sp_ai_get_household_profiles`, which explicitly excludes email and all auth data, rather than complex view hierarchies.

**Trade-off:** Every new operation requires a stored procedure. Ad-hoc queries from the application layer are not possible. This is acceptable and intentional — it keeps domain logic centralised and access rules enforced in one place.

---

## mTLS as a temporary perimeter while pre-production

**Decision:** Every external request must carry a per-user P12 client certificate, validated by Nginx before the request reaches the backend.

**Why:** The application has not yet undergone a security review. mTLS provides a hard perimeter that keeps it effectively hidden from the public internet in the interim. It is not the long-term auth strategy and will be removed once the application has been properly reviewed and a production-grade auth layer is in place.
