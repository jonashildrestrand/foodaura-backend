# Database Security

## Overview

Database access is split between two MariaDB users with separate privileges. Neither user has direct `SELECT`, `INSERT`, `UPDATE`, or `DELETE` on any table — all access goes through stored procedures. Stored procedures run with `SQL SECURITY DEFINER`, meaning they execute as the admin user who created them and have full table access internally. This enforces both row-level and column-level security at the database layer regardless of what the calling service does.

---

## Database Users

| User | Used by | Privileges |
|---|---|---|
| `foodaura_backend` | Go backend service | `EXECUTE` on all stored procedures |
| `foodaura_ai` | AI service | `EXECUTE` on AI-specific read-only stored procedures only |

Neither user can query tables directly. All row scoping and column filtering is enforced inside the stored procedures themselves.

---

## Row-Level Security

Row-level security is enforced inside stored procedures by scoping every query through `household_members`. A user can only read or modify data that belongs to a household they are a member of. The calling service passes a `requesting_user_id` (or `household_id` for AI procedures), and the procedure validates membership before returning any data.

The database user model reinforces this: because neither `foodaura_backend` nor `foodaura_ai` has direct table access, they cannot construct arbitrary queries that bypass household scoping.

---

## Column-Level Security

The AI service accesses user data only through `sp_ai_get_household_profiles`, which explicitly excludes email and any other personally identifying information not required for meal planning. The AI never has access to the `users` table, `sessions`, `household_invitations`, `notifications`, or `notification_dispatches`.

The backend accesses all data through stored procedures that are scoped to the requesting user's household. It has no mechanism to retrieve data from households the user does not belong to.

---

## AI Service — Accessible Stored Procedures

The `foodaura_ai` user has `EXECUTE` only on the following procedures. All are read-only.

| Procedure | Purpose |
|---|---|
| `sp_ai_get_household_profiles` | Nutritional profiles and targets for all members of a household — no email or auth data |
| `sp_recipe_find` | Filtered recipe catalogue |
| `sp_recipe_get` | Full recipe detail including ingredients |
| `sp_mealplan_get` | Current meal plan structure with slots and participants |

The AI service does not write to the database. All writes (persisting a generated meal plan) are performed by the backend after the AI returns its result.

---

## Stored Procedure Security Model

All stored procedures are created with `SQL SECURITY DEFINER`. This means:

- The procedure runs with the privileges of the user who **created** it (the admin), not the user who **calls** it.
- `foodaura_backend` and `foodaura_ai` can execute procedures they have no other privileges for.
- The procedure itself enforces all access rules — the calling service cannot influence what rows or columns are returned beyond the parameters it passes.

This is MariaDB's standard pattern for row-level and column-level access control without native RLS support.
